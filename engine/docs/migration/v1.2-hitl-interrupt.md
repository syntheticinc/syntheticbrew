# Engine 1.2.0 — HITL Interrupt Primitive migration

Engine 1.2.0 replaces the ad-hoc `show_structured_output` → `tool_call` SSE
path with a first-class HITL Interrupt Primitive. This is a breaking SSE
contract change for any client that today renders interactive widgets from
the `tool_call` event's `arguments` field. This doc walks through the diff
and the minimal client patch.

## What changed (TL;DR)

| | 1.1.x (before) | 1.2.0 (after) |
|---|---|---|
| Widget arrives via | `tool_call` event with `tool: "show_structured_output"` and widget schema inside `arguments` | New `interrupt_request` SSE event with envelope `{interrupt_id, kind, schema}` |
| Widget closes via | `tool_result` event (also for `show_structured_output`) | NOT emitted; client uses `interrupt_resume` event instead |
| User submit goes back via | `POST /chat {message: "<raw value>"}` — forges a fake user turn | `POST /chat {resume_interrupt: {interrupt_id, payload: {answers: [...]}}}` |
| Engine echoes the submit as | `user_message` event (caused the chirp 2026-05-20 duplicate-bubble bug) | `interrupt_resume` event with the same `interrupt_id` so clients mark the widget answered without creating a chat bubble |

`message` and `resume_interrupt` are mutually exclusive in the POST body. The
existing plain-message path is unchanged.

## New SSE events

### `interrupt_request`

Emitted whenever the agent halts at a HITL primitive (today: only
`show_structured_output`; the protocol is extensible to file picker, voice,
multi-step wizard, custom MCP HITL).

```
event: interrupt_request
data: {
  "interrupt_id": "i-7f3a-...uuid",
  "content": "{\"interrupt_id\":\"i-7f3a-...\",\"kind\":\"structured_output\",\"schema\":{\"output_type\":\"form\",\"title\":\"Configure project\",\"questions\":[{\"id\":\"platform\",\"label\":\"Target platform?\",\"type\":\"select\",\"options\":[{\"label\":\"iOS\"},{\"label\":\"Android\"}]}]}}"
}
```

Wire shape: `interrupt_id` surfaces both at the envelope level (for fast
correlation) and inside the `content` JSON (so the same string round-trips
through replay/history). `content` decodes to `InterruptRequestPayload`:

```ts
interface InterruptRequestPayload {
  interrupt_id: string;
  kind: "structured_output";   // future: "file_pick" | "voice_capture" | ...
  schema: InterruptSchema;
}

interface InterruptSchema {
  output_type: "summary_table" | "form" | "info";
  title?: string;
  description?: string;
  rows?:    { label: string; value: string }[];
  actions?: { label: string; type: "primary" | "secondary"; value: string }[];
  questions?: {
    id: string;
    label: string;
    type: "text" | "select" | "multiselect";
    options?: { label: string; value?: string }[];
    default?: string;
  }[];
}
```

### `interrupt_resume`

Echoed back after a successful `resume_interrupt` POST. Tells the client
"this widget has been answered — render the user's choice as the widget's
new state, do NOT add a chat bubble".

```
event: interrupt_resume
data: {
  "interrupt_id": "i-7f3a-...uuid",
  "content": "{\"interrupt_id\":\"i-7f3a-...\",\"kind\":\"structured_output\",\"payload\":{\"answers\":[{\"question_id\":\"platform\",\"value\":\"iOS\",\"label\":\"iOS\"}]}}"
}
```

`content` decodes to `InterruptResumePayload`:

```ts
interface InterruptResumePayload {
  interrupt_id: string;
  kind: "structured_output";
  payload: { answers: InterruptAnswer[] };
}

interface InterruptAnswer {
  question_id: string;
  value: string;
  label?: string;
}
```

## New POST body field

```
POST /api/v1/schemas/{name}/chat

// EITHER plain message (unchanged path):
{
  "session_id": "...",
  "message": "any plain user text"
}

// OR resume the most recent HITL halt:
{
  "session_id": "...",
  "resume_interrupt": {
    "interrupt_id": "i-7f3a-...uuid",
    "payload": {
      "answers": [
        {"question_id": "platform", "value": "iOS",     "label": "iOS"},
        {"question_id": "name",     "value": "MyApp"}
      ]
    }
  }
}
```

Both fields populated → 400. Neither populated → 400.

## Error mapping

| HTTP | Cause |
|---|---|
| 400 | malformed body, both `message`+`resume_interrupt` set, neither set, no session_id on resume |
| 401 | unauthenticated |
| 403 | `interrupt_id` belongs to a different session_id on the same tenant |
| 404 | unknown `interrupt_id` OR cross-tenant id (NotFound — no existence leak) |
| 409 | `interrupt_id` already resolved or abandoned, or concurrent resume race |

## Minimum client patch

For an existing client that was rendering widgets from `tool_call`
arguments and submitting via forged plain text, the three changes are:

```diff
- // 1. Listen for tool_call event with tool="show_structured_output":
- case 'tool_call':
-   if (data.tool === 'show_structured_output') {
-     const schema = JSON.parse(data.arguments);
-     renderWidget(schema);
-   }
+ // 1. Listen for interrupt_request event:
+ case 'interrupt_request':
+   const req = JSON.parse(data.content);
+   renderWidget(req.interrupt_id, req.schema);

- // 2. On user click, send the raw value as a plain message:
- function onSubmit(value) {
-   fetch('/chat', { body: JSON.stringify({ message: value }) });
- }
+ // 2. On user click, send resume_interrupt with structured answers:
+ function onSubmit(interruptId, answers) {
+   fetch('/chat', {
+     body: JSON.stringify({
+       resume_interrupt: {
+         interrupt_id: interruptId,
+         payload: { answers }
+       }
+     })
+   });
+ }

+ // 3. Listen for interrupt_resume to mark the widget answered (NO bubble):
+ case 'interrupt_resume':
+   const res = JSON.parse(data.content);
+   markWidgetAnswered(res.interrupt_id, res.payload.answers);
```

## What the LLM sees on resume

The engine reconstructs a natural-language user message from the widget
schema + submitted answers, preferring human labels over raw codes. For a
form with question label "Критичность" and selected option label "Холодовая
цепь", the React loop's next user turn carries:

```
User submitted the form:
Q: Критичность? A: Холодовая цепь
```

This is fed as a regular user message into the next LLM call — the model
sees a normal user turn, not a structured payload. No prompt change is
required on the agent side.

## Server-side details (read-only — informative)

- Server-issued `interrupt_id` is a UUID generated by `show_structured_output`
  inside the React loop. The event row stored in `session_event_log` carries
  it on the `call_id` proto field plus inside the JSON payload of `content`.
- The `interrupts` table is a pure state tracker; kind/schema/payload all
  live in the linked `session_event_log` rows reached through
  `request_event_id` and `resolve_event_id` FK. Zero duplication.
- Status transitions:
  - `pending` → set on insert when `interrupt_request` event is persisted
  - `resolved` → atomic conditional UPDATE (`WHERE status='pending'`) when
    `resume_interrupt` succeeds — second resume race → 409
  - `abandoned` → set when a regular `message` POST arrives in the same
    session while the interrupt is still pending (defensive cleanup)
